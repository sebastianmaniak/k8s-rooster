Scenario 1: Crash Loop Diagnosis

1

User sends prompt

"There's a pod crash-looping in finance-payments. Diagnose the root cause."

2

Agent plans tool calls

LLM receives system prompt with crash diagnosis instructions. Decides to call k8s_get_resources to find unhealthy pods.

3

MCP: Get pods in namespace

k8s_get_resources(kind=Pod, namespace=finance-payments) → finds payment-service in CrashLoopBackOff

4

MCP: Get events + logs

k8s_get_events shows OOMKilled restarts. k8s_get_pod_logs returns full startup output with Java stack trace.

5

Agent analyzes logs

LLM scans through 30+ log lines, finds NullPointerException at PaymentProcessor.java:314 caused by OutOfMemoryError: Java heap space.

6

Response to user

Root cause: OOM during cache warming (247K transactions into 64Mi limit). Recommends: increase memory limit or paginate cache loading.

Scenario 2: Migration Assessment

1

User sends prompt

"Which workloads are stateless and safe to migrate first?"

2

MCP: Query PVCs cluster-wide

k8s_get_resources(kind=PersistentVolumeClaim) → finds market-data-pvc in risk-management

3

MCP: Get all deployments

k8s_get_resources(kind=Deployment) across all namespaces. Correlates with PVC bindings.

4

Agent classifies workloads

risk-api (3 replicas, no PVC, Istio enabled) = safe. compliance-batch (2 replicas, no PVC) = safe. market-data-store (PVC bound) = not safe.

5

Response with migration table

Presents workload comparison table with risk ratings, replica counts, PVC status, and Istio injection status.

Scenario 4: Istio Troubleshooting

1

User sends prompt

"payment-service isn't receiving traffic through the mesh. Check the Istio config."

2

MCP: Get VirtualServices

k8s_get_resource_yaml(kind=VirtualService, name=payment-vs) → routes to payment-svc-v2:8080

3

MCP: Get Services

k8s_get_resources(kind=Service, namespace=finance-payments) → only payment-service exists. No payment-svc-v2.

4

Agent identifies mismatch

VirtualService destination host payment-svc-v2 does not match any Service. Root cause found.

5

Response with fix

Shows broken YAML vs corrected YAML side-by-side. Destination should be payment-service not payment-svc-v2.

Chaos Injection Flow

1

Trigger from browser or CLI

Click "Crash Loop" on the chaos control panel, or run ./chaos.sh crash from terminal.

2

GitHub Actions dispatches workflow

Browser calls GitHub API → chaos.yml workflow runs on self-hosted runner with kubectl access.

3

ConfigMap / VirtualService swapped

Healthy ConfigMap replaced with crashing version. Pod restarts, enters CrashLoopBackOff within ~15 seconds.

4

Demo the diagnosis

Switch to kagent UI, paste the prompt, watch the agent diagnose the issue live.
