Security Best Practices

The agent enforces these security principles when managing firewall rules.

Deny by Default

Never create policies with srcaddr: all + dstaddr: all + action: accept. Only open what's needed.

Specific Address Objects

Always use named address objects rather than "all" where possible. Group related addresses for cleaner policy sets.

Logging & Audit

Log all policy hits for audit trails. Enable logging on both accept and deny actions for full visibility.

Policy Ordering

FortiGate evaluates policies top-down. Place more specific rules before general ones. Order matters.

Example Prompts

> Audit all firewall policies and flag any that use srcaddr "all" with action "accept" — those are overly permissive.
> Create a DNAT rule to forward external port 8443 to internal host 172.16.10.186 port 443.
> List all VIPs and show me which internal hosts they map to.
> Show me all firewall policies that reference the "internal" interface as the source.
> Create an address group called "compliance-servers" containing the three compliance portal IPs, then create a policy allowing HTTPS traffic to them.

Troubleshooting Guidelines

1. List Policies

Start by listing all policies, VIPs, and interfaces. Get the lay of the land.

2. Check Ordering

FortiGate evaluates top-down. A broad deny above a specific allow will block traffic.

3. Verify Addresses

Ensure source/destination address objects exist before referencing them in policies.

4. Check VIP Mappings

Verify external IP/port maps to the correct internal IP/port. Mismatched mappings are a common issue.
