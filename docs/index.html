What is kagent?

kagent is a Kubernetes-native runtime that extends the platform with identity, orchestration, and governance for production AI agents. Rather than building agents in isolation, it provides the infrastructure to deploy them reliably at scale.

🔒

Agent Identity & Policy

Granular access control with agent-on-behalf-of semantics. Every action is governed by defined policies — agents never operate with unrestricted permissions.

🔍

Full Observability

Trace every agent action, tool call, and decision path with automatic OpenTelemetry integration. See exactly what the agent did and why.

🛠

MCP Tool Servers

Deploy, secure, and manage Model Context Protocol tool servers. Wire any internal API — Prometheus, GitHub, ServiceNow, Ansible — as agent tools.

🤝

Human-in-the-Loop

Agents propose, humans approve. Configurable checkpoints and escalation paths for operations that need oversight.

🌍

Multi-Framework

LangChain, CrewAI, Google ADK, A2A, and custom implementations — all under one governance layer.

📦

Prebuilt Agents

Production-ready agents for Kubernetes management, Helm operations, observability, and more. Start solving problems on day one.

Demo Architecture

The bank-platform-agent orchestrates specialist sub-agents (k8s, Istio, Helm, GitHub Issues) and queries Prometheus/Grafana directly. k8s-agent uses Claude Sonnet 4.6 with human-in-the-loop approval for destructive operations. Demo Scenarios

Seven scenarios that show kagent solving real platform engineering problems. Chaos is injected on demand — everything starts healthy.

1

Crash Loop Diagnosis  chaos.sh crash

A payment-service pod enters CrashLoopBackOff with a Java OOM buried in noisy startup logs. kagent reads the pod events, scans through the full log output, and surfaces the NullPointerException + OutOfMemoryError at PaymentProcessor.java:314 — distinguishing the symptom from the root cause.

> There's a pod crash-looping in the finance-payments namespace. Can you diagnose what's going wrong and tell me the root cause?
2

Migration Candidate Identification  no chaos needed

kagent queries PVCs cluster-wide, correlates with pods and deployments, identifies risk-api and compliance-batch as stateless migration candidates, and flags market-data-store (has PVC) as stateful — not safe to migrate first.

> Which workloads across all namespaces are stateless and would be safe to migrate first?
3

Cluster Capacity Forecasting  prometheus

kagent queries Prometheus via the prometheus-mcp tool server, analyses kube-state-metrics and node resource data, calculates current allocation percentages, and projects when CPU or memory will be exhausted based on growth trends.

> Based on current resource consumption trends, when will this cluster run out of allocatable CPU or memory?
4

Istio Troubleshooting  chaos.sh istio

A VirtualService routes traffic to payment-svc-v2 — a service that doesn't exist. kagent checks VirtualServices, DestinationRules, and Services, finds the mismatch, and shows the corrected YAML.

> The payment-service in finance-payments isn't receiving any traffic through the mesh. Can you check the Istio configuration?
5

Extensibility  narration only

kagent supports custom MCP tool servers — any internal API (migration APIs, Ansible Tower, ServiceNow) can be wired as an agent tool. The flow: agent identifies candidate, human approves, API executes.

6

Autonomous Incident Response with khook  chaos.sh khook

khook is a Kubernetes controller that watches cluster events and auto-triggers kagent agents — no human prompt needed. A Hook CR in compliance-ops watches for pod-restart events. When chaos.sh khook breaks the compliance-report-generator (database connection failure), khook detects the restart, creates a kagent session via A2A, and the compliance-incident-agent (Claude Sonnet 4.6) diagnoses the crash autonomously. This dedicated agent has read-only k8s tools and can create GitHub issues to track incidents.

# No human prompt — khook auto-triggers the agent
$ ./chaos.sh khook
# Watch kagent UI — agent starts diagnosing within seconds
event-driven autonomous A2A protocol
7

Ambient Mesh Security Lockdown  chaos.sh mesh

Istio ambient mesh protects all demo namespaces with ztunnel L4 encryption (no sidecars). chaos.sh mesh injects a STRICT PeerAuthentication + deny-all AuthorizationPolicy on finance-payments, blocking all traffic to payment-service. kagent diagnoses the security policy misconfiguration and shows the fix.

> The payment-service in finance-payments is rejecting all connections. Can you check the Istio security policies and figure out what's blocking traffic?
ambient mesh security policy ztunnel L4

Running the Demo

Two scripts manage the entire lifecycle. Deploy healthy, inject chaos on demand, reset when done.

demo/bank
# Deploy everything in a healthy state
$ ./setup.sh apply
# Verify all pods running, agents registered
$ ./setup.sh status
# Inject chaos for Scenario 1 (Java OOM crashloop)
$ ./chaos.sh crash
# Inject chaos for Scenario 4 (broken Istio routing)
$ ./chaos.sh istio
# Inject chaos for Scenario 6 (khook auto-response)
$ ./chaos.sh khook
# Inject chaos for Scenario 7 (break ambient mesh)
$ ./chaos.sh mesh
# Inject all chaos at once
$ ./chaos.sh all
# Restore everything to healthy
$ ./chaos.sh reset
# Tear down the entire demo
$ ./setup.sh delete
Command What it does
./setup.sh apply Create namespaces, deploy healthy workloads, register agent + Prometheus MCP
./setup.sh status Show all pods, agents, MCP servers, VirtualServices, quotas
./chaos.sh crash Swap payment-service ConfigMap to crashing version, trigger CrashLoopBackOff
./chaos.sh istio Patch VirtualService to route to non-existent payment-svc-v2
./chaos.sh khook Break compliance-report-generator — khook auto-triggers agent diagnosis
./chaos.sh mesh Inject STRICT mTLS + deny-all AuthorizationPolicy — blocks mesh traffic
./chaos.sh all Inject crash + Istio + khook + mesh failures at once
./chaos.sh reset Restore healthy ConfigMap, correct VirtualService, restart pods
./setup.sh delete Remove all demo resources including namespaces

Deployed Agents

Six AI agents in the kagent namespace, each with specialized tools and A2A skills. Three LLM backends: Claude Sonnet 4.6, xAI Grok, and OpenAI via AgentGateway.

k8s-agent

Claude Sonnet 4.6

Kubernetes expert with human-in-the-loop approval for destructive operations. Manages cluster ops, troubleshooting, and maintenance.

Tools: kagent-tool-server (18 tools)
Skills: cluster-diagnostics, resource-management, security-audit

compliance-incident-agent

Claude Sonnet 4.6

Autonomous incident responder triggered by khook on pod restarts and OOM kills. Diagnoses crashes, creates GitHub issues.

Tools: kagent-tool-server (7 read-only tools)
Skills: compliance-incident-response
Delegates to: github-issues-agent

github-issues-agent

xAI Grok

Manages GitHub repos — creates issues, branches, PRs, and file updates. Used by other agents for incident tracking.

Tools: github-mcp-remote (18 tools)
Skills: manage-github-repository

github-pr-agent

AgentGateway LLM

Reviews and fixes pull requests. Code review, branch management, and Copilot review integration.

Tools: github-mcp-remote (16 tools)
Skills: fix-pull-requests

f5-bigip-agent

xAI Grok

F5 BIG-IP network infrastructure — manages pools, virtual servers, nodes, and health monitors via iControl REST. HITL required for all changes.

Tools: f5-bigip-mcp (17 tools)
Skills: f5-bigip-management
Creds: Vault → f5-bigip-credentials

fortigate-agent

xAI Grok

FortiGate firewall management — policies, VIPs, addresses, address groups, and interfaces via FortiOS REST. HITL required for all changes.

Tools: fortigate-mcp (16 tools)
Skills: fortigate-management
Creds: Vault → fortigate-credentials
