kagent Demo Platform

AI-powered Kubernetes operations. Crash diagnosis, migration planning, Istio ambient mesh troubleshooting, capacity forecasting, and Ansible infrastructure management — driven by an orchestrator agent with specialist sub-agents.

What is kagent?

kagent is a Kubernetes-native runtime that extends the platform with identity, orchestration, and governance for production AI agents. Rather than building agents in isolation, it provides the infrastructure to deploy them reliably at scale.

🔒

Agent Identity & Policy

Granular access control with agent-on-behalf-of semantics. Every action is governed by defined policies — agents never operate with unrestricted permissions.

🔍

Full Observability

Trace every agent action, tool call, and decision path with automatic OpenTelemetry integration. See exactly what the agent did and why.

🛠

MCP Tool Servers

Deploy, secure, and manage Model Context Protocol tool servers. Wire any internal API — Prometheus, GitHub, ServiceNow, Ansible — as agent tools.

🤝

Human-in-the-Loop

Agents propose, humans approve. Configurable checkpoints and escalation paths for operations that need oversight.

🌍

Multi-Framework

LangChain, CrewAI, Google ADK, A2A, and custom implementations — all under one governance layer.

📦

Prebuilt Agents

Production-ready agents for Kubernetes management, Helm operations, observability, and more. Start solving problems on day one.

Kubernetes

Cluster operations, workload management, incident response, and observability

k8s-agent slackbot-k8s-agent compliance-incident-agent github-issues-agent github-pr-agent

Automated Incident Response

A pod crashes at 3 AM. khook detects the restart, triggers compliance-incident-agent which reads logs, identifies the root cause (e.g. database connection timeout), and delegates to github-issues-agent to file a detailed incident report — all without a human touching a keyboard.

Flow: khook → compliance-incident-agent → github-issues-agent

Live Namespace Migration

Move a running application from one namespace to another with zero downtime. k8s-agent exports all resources (Deployments, Services, ConfigMaps), recreates them in the target namespace, verifies health, and cleans up the source — with human approval at each step.

Flow: k8s-agent (HITL)

Slack-Driven Cluster Ops

Ask questions in Slack and get instant answers: "Which pods are unhealthy?" or "What Helm releases are failing?". slackbot-k8s-agent queries the cluster and posts formatted reports directly to your channel.

Flow: slackbot-k8s-agent → Slack

Crash Loop Diagnosis

A payment-service pod enters CrashLoopBackOff with a Java OOM buried in noisy logs. k8s-agent reads pod events, scans full log output, and surfaces the root cause — distinguishing the symptom (OOM kill) from the bug (NullPointerException at PaymentProcessor.java:314).

Flow: k8s-agent (read-only diagnosis)

Networking

Service mesh, load balancing, Gateway API, and traffic management

istio-agent f5-bigip-agent kgateway-agent k8s-f5-agent

Istio Traffic Splitting & Canary

Route 10% of traffic to a new version of your service for canary testing. istio-agent creates VirtualService and DestinationRule resources, monitors proxy status, and can roll back instantly if errors spike.

Flow: istio-agent (HITL)

End-to-End Service Exposure

Deploy a new service and expose it through the F5 load balancer in one conversation. k8s-f5-agent checks the K8s deployment is healthy, grabs the NodePort, creates an F5 pool with health monitors, and binds a virtual server — full stack from pod to public VIP.

Flow: k8s-f5-agent (HITL)

Gateway API Configuration

Expose services through Kubernetes Gateway API with kgateway-agent. Create Gateway resources, configure HTTPRoutes with path-based routing, set up TLS, and manage traffic policies — all from natural language.

Flow: kgateway-agent (HITL)

Ambient Mesh Security Lockdown

Istio ambient mesh protects all namespaces with ztunnel L4 encryption (no sidecars). When a STRICT PeerAuthentication + deny-all AuthorizationPolicy blocks traffic, istio-agent diagnoses the security policy misconfiguration and shows the fix.

Flow: istio-agent (diagnosis + HITL fix)

Security

Firewall policies, network policies, secrets management, and PKI

fortigate-agent cilium-crd-agent vault-agent

Firewall Policy Audit

Ask fortigate-agent to audit all firewall policies and flag overly permissive rules (e.g. srcaddr: all with action: accept). It lists every policy, identifies violations of deny-by-default best practices, and can tighten rules with your approval.

Flow: fortigate-agent (HITL)

Zero-Trust Network Policy

Describe your security intent: "Only frontend pods should reach the API on port 8080". cilium-crd-agent generates valid CiliumNetworkPolicy YAML with the correct selectors, ports, and L7 rules, then applies it after you review.

Flow: cilium-crd-agent (HITL)

Secrets & Certificate Management

Set up a full PKI chain in HashiCorp Vault: create a root CA, an intermediate CA, configure roles, and issue leaf certificates with proper SANs and TTLs. vault-agent masks all secret values and requires approval for every write.

Flow: vault-agent (HITL)

DNAT & Port Forwarding

Create destination NAT rules to forward external traffic to internal services. fortigate-agent configures VIP objects, binds them to firewall policies, and verifies the forwarding path — with full audit trail and human approval.

Flow: fortigate-agent (HITL)

Infrastructure Fix via GitOps

The Compliance Portal is returning 502. ansible-infra-agent reads the Ansible inventory from GitHub, spots the misconfigured upstream IP, and submits a PR via github-pr-agent to fix it — infrastructure-as-code with AI-driven remediation.

Proxmox LXC Provisioning

Provision a 3-tier application stack on Proxmox LXC containers via Ansible. ansible-infra-agent manages the full lifecycle — create containers, deploy Nginx + Flask + PostgreSQL, configure networking, and run health checks.

Flow: ansible-infra-agent (Semaphore API)

Automated PR Workflows

Review pull requests, run Copilot code review, and fix issues automatically. github-pr-agent reads diffs, identifies problems, creates fix commits, and manages branch workflows — integrated with GitHub's native tooling.

Flow: github-pr-agent (AgentGateway LLM)

Compliance Scanning & Reporting

Run compliance scans across infrastructure, generate reports, and automatically file GitHub issues for violations. ansible-infra-agent coordinates the scan and delegates issue tracking to github-issues-agent.

Flow: ansible-infra-agent → github-issues-agent

Demo Scenarios

Ten scenarios that show kagent solving real platform engineering problems. Chaos is injected on demand — everything starts healthy.

1

Crash Loop Diagnosis  chaos.sh crash

A payment-service pod enters CrashLoopBackOff with a Java OOM buried in noisy startup logs. kagent reads the pod events, scans through the full log output, and surfaces the NullPointerException + OutOfMemoryError at PaymentProcessor.java:314 — distinguishing the symptom from the root cause.

> There's a pod crash-looping in the finance-payments namespace. Can you diagnose what's going wrong and tell me the root cause?
2

Migration Candidate Identification  no chaos needed

kagent queries PVCs cluster-wide, correlates with pods and deployments, identifies risk-api and compliance-batch as stateless migration candidates, and flags market-data-store (has PVC) as stateful — not safe to migrate first.

> Which workloads across all namespaces are stateless and would be safe to migrate first?
3

Cluster Capacity Forecasting  prometheus

kagent queries Prometheus via the prometheus-mcp tool server, analyses kube-state-metrics and node resource data, calculates current allocation percentages, and projects when CPU or memory will be exhausted based on growth trends.

> Based on current resource consumption trends, when will this cluster run out of allocatable CPU or memory?
4

Istio Troubleshooting  chaos.sh istio

A VirtualService routes traffic to payment-svc-v2 — a service that doesn't exist. kagent checks VirtualServices, DestinationRules, and Services, finds the mismatch, and shows the corrected YAML.

> The payment-service in finance-payments isn't receiving any traffic through the mesh. Can you check the Istio configuration?
5

Extensibility  narration only

kagent supports custom MCP tool servers — any internal API (migration APIs, Ansible Tower, ServiceNow) can be wired as an agent tool. The flow: agent identifies candidate, human approves, API executes.

6

Autonomous Incident Response with khook  chaos.sh khook

khook is a Kubernetes controller that watches cluster events and auto-triggers kagent agents — no human prompt needed. A Hook CR in compliance-ops watches for pod-restart events. When chaos.sh khook breaks the compliance-report-generator (database connection failure), khook detects the restart, creates a kagent session via A2A, and the compliance-incident-agent (Claude Sonnet 4.6) diagnoses the crash autonomously. This dedicated agent has read-only k8s tools and can create GitHub issues to track incidents.

# No human prompt — khook auto-triggers the agent
$ ./chaos.sh khook
# Watch kagent UI — agent starts diagnosing within seconds
event-driven autonomous A2A protocol
7

Ambient Mesh Security Lockdown  chaos.sh mesh

Istio ambient mesh protects all demo namespaces with ztunnel L4 encryption (no sidecars). chaos.sh mesh injects a STRICT PeerAuthentication + deny-all AuthorizationPolicy on finance-payments, blocking all traffic to payment-service. kagent diagnoses the security policy misconfiguration and shows the fix.

> The payment-service in finance-payments is rejecting all connections. Can you check the Istio security policies and figure out what's blocking traffic?
ambient mesh security policy ztunnel L4
8

Ansible Infrastructure Troubleshooting  proxmox

The Compliance Portal runs on 3 Proxmox LXC containers managed by Ansible (Nginx → Flask → PostgreSQL). A misconfigured upstream IP in the Ansible inventory causes the portal to return 502 Bad Gateway. The ansible-infra-agent reads the Ansible code from GitHub, identifies the wrong nginx_upstream_host in webservers.yml, and submits a fix as a GitHub PR via github-pr-agent.

> The compliance portal at 172.16.10.186 is returning 502 Bad Gateway. Can you diagnose the issue and create a fix?
Portal URL: http://172.16.10.186
Containers: compliance-web (VMID 200) · compliance-app (VMID 201) · compliance-db (VMID 202)
Agent flow: ansible-infra-agent → reads GitHub → finds bad IP → delegates to github-pr-agent → PR created
ansible GitOps A2A delegation
9

Namespace Migration  chaos.sh migrate

A fruit-app deployment (2 replicas + Service + ConfigMap) runs in namespace apples. The k8s-agent exports all resources, recreates them in namespace oranges, verifies the app is healthy, and cleans up the source namespace — a full live migration with human-in-the-loop approval at each step.

> Migrate the fruit-app and all its resources from namespace apples to namespace oranges. Make sure it's running in oranges, then clean up apples.
k8s-agent HITL namespace migration
10

F5 BIG-IP Load Balancer Management  f5-bigip-agent

The f5-bigip-agent manages an F5 BIG-IP appliance via iControl REST. It can list and inspect pools, virtual servers, nodes, and health monitors. With human approval, it creates pools, adds/removes pool members, configures VIPs, and enables/disables nodes for maintenance — all through natural language.

> List all pools on the BIG-IP and show me which ones have unhealthy members.
> Disable node 172.16.20.50 for maintenance and verify it's been removed from the active pool.
F5 BIG-IP HITL iControl REST
11

FortiGate Firewall Policy Audit  fortigate-agent

The fortigate-agent manages a FortiGate firewall via FortiOS REST. It audits firewall policies, inspects VIP/DNAT rules, reviews address objects and interface assignments. With human approval, it can create policies, configure port forwarding (VIPs), and manage address objects — enforcing deny-by-default best practices.

> Audit all firewall policies and flag any that use srcaddr "all" with action "accept" — those are overly permissive.
> Create a DNAT rule to forward external port 8443 to internal host 172.16.10.186 port 443.
FortiGate HITL FortiOS REST
12

Kubernetes Operations  k8s-agent

The k8s-agent is the general-purpose Kubernetes expert. It handles cluster diagnostics, resource management, and security audits. It can describe pods, read logs, check events, inspect YAML, verify service connectivity, and with human approval — apply manifests, patch resources, scale deployments, and manage labels/annotations.

> Check all pods across the cluster and tell me which ones are in a bad state. Show me the root cause for each.
> Scale the risk-api deployment in risk-management to 5 replicas and verify all pods are ready.
Claude Sonnet 4.6 HITL 18 k8s tools

Running the Demo

Two scripts manage the entire lifecycle. Deploy healthy, inject chaos on demand, reset when done.

demo/bank
# Deploy everything in a healthy state
$ ./setup.sh apply
# Verify all pods running, agents registered
$ ./setup.sh status
# Inject chaos for Scenario 1 (Java OOM crashloop)
$ ./chaos.sh crash
# Inject chaos for Scenario 4 (broken Istio routing)
$ ./chaos.sh istio
# Inject chaos for Scenario 6 (khook auto-response)
$ ./chaos.sh khook
# Inject chaos for Scenario 7 (break ambient mesh)
$ ./chaos.sh mesh
# Set up namespace migration scenario (Scenario 9)
$ ./chaos.sh migrate
# Inject all chaos at once
$ ./chaos.sh all
# Restore everything to healthy
$ ./chaos.sh reset
# Tear down the entire demo
$ ./setup.sh delete
Command What it does
./setup.sh apply Create namespaces, deploy healthy workloads, register agent + Prometheus MCP
./setup.sh status Show all pods, agents, MCP servers, VirtualServices, quotas
./chaos.sh crash Swap payment-service ConfigMap to crashing version, trigger CrashLoopBackOff
./chaos.sh istio Patch VirtualService to route to non-existent payment-svc-v2
./chaos.sh khook Break compliance-report-generator — khook auto-triggers agent diagnosis
./chaos.sh mesh Inject STRICT mTLS + deny-all AuthorizationPolicy — blocks mesh traffic
./chaos.sh migrate Deploy fruit-app in apples namespace, create empty oranges for migration
./chaos.sh all Inject crash + Istio + khook + mesh failures at once
./chaos.sh reset Restore healthy ConfigMap, correct VirtualService, restart pods
./setup.sh delete Remove all demo resources including namespaces
