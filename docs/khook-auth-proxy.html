Server-to-Server Auth

khook Auth Proxy

How khook authenticates with kagent-enterprise to trigger AI agents from Kubernetes events — without OIDC browser flows.

The Problem

kagent-enterprise requires OIDC authentication for all API calls. khook makes direct server-to-server requests — no browser, no cookies, no OIDC tokens.

✗ Without Auth Proxy

  • khook calls POST /api/sessions directly
  • Enterprise middleware rejects — no session cookie
  • Returns HTTP 400: Failed to get user ID
  • AUTO_AUTH_ENABLED only works for browser UI
  • OIDC auth code flow requires browser redirect

✓ With Auth Proxy

  • khook calls the proxy on the same port (8083)
  • Proxy injects K8s ServiceAccount token
  • Proxy adds X-User-Id: admin header
  • Controller validates SA token via JWKS
  • Session created, agent triggered via A2A

Request Architecture

The proxy is a transparent pass-through that adds two headers. khook doesn't know it exists.

Request Flow

Step-by-step: from cluster event to agent action.

1

Cluster Event

A pod enters CrashLoopBackOff, gets OOM-killed, or triggers a restart event in a watched namespace.

2

khook Detects & Matches

khook's event watcher sees the event and matches it against configured hooks (event type, namespace, agent target). Duplicates are suppressed.

3

Proxy Injects Auth

khook sends POST /api/sessions to the proxy. The proxy reads the pod's mounted SA token (cached by file mtime), strips any existing auth headers, and injects Authorization: Bearer <token> + X-User-Id: admin.

4

Session Created

The kagent-enterprise controller validates the K8s SA token against the cluster JWKS endpoint, accepts the X-User-Id, and creates a new session (HTTP 201). Returns a contextId.

5

Agent Triggered via A2A

khook sends an A2A SendMessage with the event context (pod name, namespace, event type, timestamps). The target agent investigates and takes action.

Project Files

Three files in the khook/ directory, deployed via ArgoCD.

khook/
  kustomization.yaml — Bundles resources with common labels
  auth-proxy-script.yaml — ConfigMap: Python reverse proxy (~160 lines)
  auth-proxy-deployment.yaml — ServiceAccount + Deployment + Service

kagent/
  khook-application.yaml — ArgoCD app: points khook apiUrl at the proxy

manifests/kagent/
  khook-auth-proxy-application.yaml — ArgoCD app: deploys the proxy itself

Configuration

Environment variables control the proxy behavior. Set in auth-proxy-deployment.yaml.

Variable Default Description
UPSTREAM_URL http://kagent-controller.kagent.svc.cluster.local:8083 kagent-enterprise controller URL
USER_ID admin Injected as X-User-Id header
SA_TOKEN_PATH /var/run/secrets/kubernetes.io/serviceaccount/token Path to the mounted ServiceAccount token
LISTEN_PORT 8083 Port the proxy listens on

khook Helm Values

The khook ArgoCD application overrides kagent.apiUrl to route through the proxy.

# kagent/khook-application.yaml
spec:
  source:
    helm:
      valuesObject:
        kagent:
          # Route through auth proxy instead of directly to controller
          apiUrl: http://khook-auth-proxy.kagent.svc.cluster.local:8083
          userId: admin@kagent.dev

Proxy Internals

A ~160-line Python script using only the standard library. No external dependencies.

SA Token Caching

Reads the token file once and caches it in memory. Re-reads only when the file's mtime changes (K8s rotates projected tokens periodically).

Header Injection

Strips incoming Authorization, X-User-Id, Host, and Transfer-Encoding. Injects the SA bearer token and configured user ID.

Streaming Response

Reads upstream responses in 8KB chunks and streams them back. Handles both success and HTTP error responses from the controller.

Health Endpoint

Separate HTTP server on port 8084 serves /healthz. Used by K8s readiness (3s/5s) and liveness (5s/15s) probes.

All HTTP Methods

Proxies GET, POST, PUT, PATCH, DELETE, and OPTIONS. Covers the full kagent API surface (sessions, A2A, agents).

Resource Footprint

CPU: 25m request / 100m limit. Memory: 32Mi request / 64Mi limit. Single replica. Uses python:3.12-slim base image.

Troubleshooting

Quick diagnostic commands when things aren't working.

diagnostic commands
# Check proxy pod status $ kubectl get pods -n kagent -l app=khook-auth-proxy # View proxy logs (look for "Proxy ready" and "SA token loaded") $ kubectl logs -n kagent -l app=khook-auth-proxy # Verify SA token is mounted and non-empty $ kubectl exec -n kagent deploy/khook-auth-proxy -- \ wc -c /var/run/secrets/kubernetes.io/serviceaccount/token # Test proxy health endpoint $ kubectl exec -n kagent deploy/khook-auth-proxy -- \ python3 -c "import urllib.request; print(urllib.request.urlopen('http://localhost:8084/healthz').read())" # Check that khook is pointed at the proxy (not controller) $ kubectl get cm khook-config -n kagent -o jsonpath='{.data.config\.yaml}' | grep apiUrl # Check khook logs for session creation $ kubectl logs -n kagent -l app.kubernetes.io/name=khook --tail=50
Symptom Cause Fix
HTTP 400 — "no session found" khook is hitting the controller directly, bypassing the proxy Check kagent.apiUrl in khook helm values points to khook-auth-proxy
HTTP 401 — "unauthorized" SA token is empty or expired Check the SA token file is mounted and non-empty. Verify the ServiceAccount exists
HTTP 403 — "permission denied" SA token is valid but missing the X-User-Id header Verify the USER_ID env var is set in the proxy deployment
HTTP 502 — "upstream error" Proxy can't reach the kagent controller Check UPSTREAM_URL env var and that kagent-controller service exists
Proxy pod in CrashLoopBackOff Python script error or missing ConfigMap mount Check that ConfigMap khook-auth-proxy-script exists and is mounted at /app

Why Not OIDC?

We investigated every OIDC path before landing on the SA token approach.

Approach Result Why It Failed
Failed  OIDC client_credentials invalid_client Auto IDP has no clients configured for client_credentials grant
Failed  OIDC password grant not supported Auto IDP explicitly disables the password grant type
Failed  OIDC auth code flow Connection dropped The /authorize endpoint only responds to browser JavaScript, drops non-browser TCP connections
Failed  Self-signed JWT HTTP 401 Controller validates API tokens against the OIDC issuer JWKS, not its own signing key
Failed  SA token alone HTTP 403 Token authenticates but SA has no OIDC Groups claim, so RBAC maps to no role
Works  SA token + X-User-Id HTTP 201 Combined headers bypass OIDC session requirement for API session creation
